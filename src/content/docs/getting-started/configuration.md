# Configuration

Your agent's behavior is controlled by `forge.yaml`, the central configuration file generated by `forge init`. Every section is fully editable. This guide covers each section in detail.

## Complete Example

Here is a fully annotated `forge.yaml` showing all available sections:

```yaml
agent_id: my-agent
version: 0.1.0
framework: forge                   # forge (default) | crewai | langchain â€” "custom" accepted as alias
registry: ghcr.io/org              # Container registry (optional)
entrypoint: agent.py               # Required for crewai/langchain, omit for forge

model:
  provider: openai
  name: gpt-4o
  organization_id: "org-xxx"       # OpenAI Organization ID (enterprise, optional)
  fallbacks:
    - provider: anthropic
      name: claude-sonnet-4-20250514
    - provider: gemini
      name: gemini-2.5-flash

memory:
  persistence: true
  sessions_dir: .forge/sessions
  trigger_ratio: 0.6
  char_budget: 200000
  long_term: true
  memory_dir: .forge/memory
  embedding_provider: openai
  embedding_model: ""
  vector_weight: 0.7
  keyword_weight: 0.3
  decay_half_life_days: 7

skills:
  path: SKILL.md

tools:
  - name: cli_execute
    config:
      allowed_binaries: [summarize, curl]
      timeout: 120s
  - name: web_search
    config:
      provider: tavily
  - name: http_request
  - name: json_parse

channels:
  - slack
  - telegram

secrets:
  providers:
    - encrypted-file
    - env

egress:
  profile: standard
  mode: allowlist
  capabilities:
    - slack
    - telegram
  allowed_domains:
    - custom-api.example.com
    - "*.github.com"

schedules:
  - id: daily-report
    cron: "@daily"
    task: "Generate and send the daily status report"
    skill: "tavily-research"       # optional: invoke a specific skill
    channel: telegram              # optional: deliver results to a channel
    channel_target: "-100123456"   # optional: destination chat/channel ID
```

The sections below walk through each block.

## Top-Level Fields

| Field | Type | Description |
|---|---|---|
| `agent_id` | string | Unique identifier for your agent. Auto-generated from the name during `forge init`. |
| `version` | string | Semantic version of your agent configuration. |
| `framework` | string | Runtime framework. Defaults to `forge`. Accepts `forge`, `crewai`, `langchain`, or `custom` (alias for `forge`). |
| `registry` | string | Container registry prefix for `forge package` (e.g., `ghcr.io/org`). Optional. |
| `entrypoint` | string | Entry point script. Required for `crewai`/`langchain` frameworks, omit for `forge`. |

## Model Providers

The `model` block configures your primary LLM provider and optional fallback chain.

```yaml
model:
  provider: openai
  name: gpt-4o
  organization_id: "org-xxxxxxxxxxxxxxxxxxxxxxxx"  # OpenAI enterprise (optional)
  fallbacks:
    - provider: anthropic
      name: claude-sonnet-4-20250514
    - provider: gemini
      name: gemini-2.5-flash
```

### Supported Providers

| Provider | Example Model | Env Var | Notes |
|---|---|---|---|
| `openai` | `gpt-5.2-2025-12-11` | `OPENAI_API_KEY` | Default provider in the init wizard |
| `anthropic` | `claude-sonnet-4-20250514` | `ANTHROPIC_API_KEY` | Supports extended thinking |
| `gemini` | `gemini-2.5-flash` | `GEMINI_API_KEY` | Google AI Studio or Vertex AI |
| `ollama` | `llama3` | None | Local inference, no API key needed |

### Fallback Chains

Fallbacks are tried in order when the primary provider returns an error or is unreachable. Each fallback entry requires a `provider` and `name`.

```yaml
fallbacks:
  - provider: anthropic
    name: claude-sonnet-4-20250514
  - provider: gemini
    name: gemini-2.5-flash
```

If all providers fail, the agent surfaces the error to the user. Fallback attempts are logged to help you diagnose connectivity or quota issues.

You can override the fallback chain at runtime using the `FORGE_MODEL_FALLBACKS` environment variable (comma-separated `provider:model` pairs).

## Tools

The `tools` array defines which built-in tools your agent can use. Each tool has a `name` and an optional `config` block.

```yaml
tools:
  - name: cli_execute
    config:
      allowed_binaries: [summarize, curl]
      timeout: 120s
  - name: web_search
    config:
      provider: tavily
  - name: http_request
  - name: json_parse
```

### Built-in Tools

| Tool | Description | Config Options |
|---|---|---|
| `cli_execute` | Run allowed shell commands | `allowed_binaries` (list), `timeout` (duration) |
| `web_search` | Search the web | `provider`: `tavily` or `perplexity` |
| `http_request` | Make HTTP requests | None |
| `json_parse` | Parse and query JSON | None |
| `read_skill` | Load a skill's SKILL.md on demand | None (always available) |

### Auto-Derived Tool Config

When you add a skill that declares `requires.bins` in its frontmatter, `forge build` automatically adds those binaries to `cli_execute.allowed_binaries`. You do not need to update the tools section manually for skill dependencies.

## Memory

Forge supports two layers of memory: session memory (always active) and long-term memory (opt-in).

```yaml
memory:
  persistence: true
  sessions_dir: .forge/sessions
  trigger_ratio: 0.6
  char_budget: 200000
  long_term: true
  memory_dir: .forge/memory
  embedding_provider: openai
  embedding_model: ""
  vector_weight: 0.7
  keyword_weight: 0.3
  decay_half_life_days: 7
```

### Session Memory

Session memory persists conversation history across restarts. It is always active when `persistence` is `true`.

| Field | Type | Default | Description |
|---|---|---|---|
| `persistence` | boolean | `true` | Enable session memory persistence to disk |
| `sessions_dir` | string | `.forge/sessions` | Directory where session files are stored |
| `trigger_ratio` | float | `0.6` | Context window usage ratio that triggers summarization. When the conversation fills 60% of the context window, older messages are summarized to free space. |
| `char_budget` | integer | `200000` | Maximum character budget for session context |

### Long-Term Memory

Long-term memory enables your agent to recall facts and context across separate sessions. It uses embedding-based vector search combined with keyword matching.

| Field | Type | Default | Description |
|---|---|---|---|
| `long_term` | boolean | `false` | Enable long-term memory |
| `memory_dir` | string | `.forge/memory` | Directory where memory data is stored |
| `embedding_provider` | string | `openai` | Provider for generating embeddings |
| `embedding_model` | string | `""` | Specific embedding model. Empty string uses the provider's default. |
| `vector_weight` | float | `0.7` | Weight for vector similarity in retrieval scoring |
| `keyword_weight` | float | `0.3` | Weight for keyword matching in retrieval scoring |
| `decay_half_life_days` | integer | `7` | Half-life in days for memory recency decay. Older memories score lower. |

## Channels

Channels connect your agent to messaging platforms. List the channels you want to enable:

```yaml
channels:
  - slack
  - telegram
```

Channels are started alongside the agent using `forge serve --with <channel>`. They are not separate processes. To configure a channel, use the interactive setup command:

```bash
forge channel add slack
forge channel add telegram
```

These commands prompt for the required tokens, validate them, and store them in your secrets provider.

## Egress

The egress block controls which external domains your agent can reach. This is a core security feature of Forge.

```yaml
egress:
  profile: standard
  mode: allowlist
  capabilities:
    - slack
    - telegram
  allowed_domains:
    - custom-api.example.com
    - "*.github.com"
```

### Profiles

Profiles are presets that set sensible defaults for egress behavior:

| Profile | Description |
|---|---|
| `strict` | Only domains explicitly listed in skill SKILL.md files and `allowed_domains`. No wildcards allowed. |
| `standard` | Skill-declared domains plus `allowed_domains`. Wildcard subdomains permitted. Default for production. |
| `permissive` | All skill-declared domains allowed plus broad wildcards. Suitable for experimentation only. |

### Modes

Modes control the enforcement strategy:

| Mode | Description |
|---|---|
| `deny-all` | Block all outbound traffic. Only domains from skills and `allowed_domains` are permitted. |
| `allowlist` | Default. Permit traffic to the computed allowlist (skill domains + `allowed_domains` + capability domains). |
| `dev-open` | Allow all outbound traffic. **Not allowed in production builds** â€” `forge package --prod` rejects this mode. |

### Capabilities

The `capabilities` list auto-adds the required domains for each integration. For example, listing `slack` adds the Slack API domains to the allowlist automatically.

### Inspecting the Allowlist

To see the full computed allowlist with source annotations (which skill or config contributed each domain):

```bash
forge security egress show
```

## Secrets

The `secrets` block configures how your agent stores and retrieves sensitive values like API keys and tokens.

```yaml
secrets:
  providers:
    - encrypted-file
    - env
```

### Providers

Providers are checked in order. The first provider that has the requested secret wins.

| Provider | Storage | Description |
|---|---|---|
| `encrypted-file` | `.forge/secrets.enc` | AES-256-GCM encrypted file. Requires `FORGE_PASSPHRASE` to unlock. |
| `env` | Environment variables | Falls back to reading from the process environment. |

### Managing Secrets

```bash
# Store a secret in the encrypted file
forge secret set OPENAI_API_KEY

# Retrieve a secret (shows which provider it came from)
forge secret get OPENAI_API_KEY

# List all stored keys
forge secret list
```

The encrypted file is safe to commit to version control. Without `FORGE_PASSPHRASE`, the contents are unreadable.

## Schedules

The `schedules` block configures recurring cron-based tasks. See [Scheduling](/docs/core-concepts/scheduling) for full details.

```yaml
schedules:
  - id: daily-report
    cron: "@daily"
    task: "Generate and send the daily status report"
    skill: "tavily-research"
    channel: telegram
    channel_target: "-100123456"
```

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique schedule identifier |
| `cron` | string | Cron expression (`@daily`, `*/15 * * * *`, `@every 5m`) |
| `task` | string | Task description sent to the agent |
| `skill` | string | Optional skill to invoke |
| `channel` | string | Optional channel for result delivery |
| `channel_target` | string | Destination chat/channel ID |

## Environment Variable Overrides

Several configuration values can be overridden at runtime using environment variables. This is useful for CI/CD pipelines and container deployments where you do not want to modify `forge.yaml`.

| Variable | Overrides | Example |
|---|---|---|
| `FORGE_PASSPHRASE` | Secrets encryption passphrase | `export FORGE_PASSPHRASE="my-secret-phrase"` |
| `FORGE_MODEL_PROVIDER` | `model.provider` | `FORGE_MODEL_PROVIDER=anthropic` |
| `FORGE_MEMORY_PERSISTENCE` | `memory.persistence` | `FORGE_MEMORY_PERSISTENCE=false` |
| `FORGE_MEMORY_LONG_TERM` | `memory.long_term` | `FORGE_MEMORY_LONG_TERM=true` |
| `FORGE_EMBEDDING_PROVIDER` | `memory.embedding_provider` | `FORGE_EMBEDDING_PROVIDER=gemini` |
| `FORGE_MODEL_FALLBACKS` | `model.fallbacks` | `FORGE_MODEL_FALLBACKS="anthropic:claude-sonnet-4-20250514,gemini:gemini-2.5-flash"` |
| `OPENAI_ORG_ID` | `model.organization_id` | `OPENAI_ORG_ID=org-xxx` |
| `TAVILY_API_KEY` | Tavily web search API key | `TAVILY_API_KEY=tvly-...` |
| `PERPLEXITY_API_KEY` | Perplexity web search API key | `PERPLEXITY_API_KEY=pplx-...` |
| `WEB_SEARCH_PROVIDER` | Force web search provider | `WEB_SEARCH_PROVIDER=perplexity` |
| `OPENAI_BASE_URL` | Override OpenAI base URL | `OPENAI_BASE_URL=https://custom.api.com/v1` |
| `ANTHROPIC_BASE_URL` | Override Anthropic base URL | `ANTHROPIC_BASE_URL=https://custom.api.com` |
| `OLLAMA_BASE_URL` | Override Ollama base URL | `OLLAMA_BASE_URL=http://localhost:11434` |

Environment variables take precedence over `forge.yaml` values. The original file is not modified.
